# Anti-WeChat Spec / 反微信通讯协议

&gt; 一个假设：如果通讯工具回归"工具"属性，而不是"社交表演"舞台，会是什么样子？

## 痛点清单（按崩溃程度排序）

| 场景 | 微信的表现 | 理想表现 |
|------|-----------|---------|
| 凌晨收到工作消息 | 显示"已读"，被迫立即回复 | 可设置"异步模式"，对方看到"该用户专注中，预计X小时后回复" |
| 朋友圈点赞 | 共同好友提醒，社交压力 | 无点赞功能，或点赞不通知任何人 |
| 电脑端登录 | 必须手机扫码，手机没电就失联 | 独立账号体系，多端同时在线 |
| 群聊沉没 | 信息刷屏，重要内容被淹没 | 频道-线程结构，支持话题归档 |
| 关系混杂 | 同事、家人、网友全在一个列表 | 多身份容器，完全隔离的社交图谱 |

## 核心原则

1. **异步优先**：不鼓励即时回复，尊重用户注意力
2. **身份隔离**：一个账号 = 多个面具，数据不互通
3. **无算法时间线**：按时间排序，无推荐，无"可能认识的人"
4. **默认隐私**：不显示在线状态、正在输入、已读

## 非目标（明确不做什么）

- ❌ 不做端到端加密（合规成本太高）
- ❌ 不做陌生人社交/匹配（避免变成相亲软件）
- ❌ 不做短视频/直播（不是内容平台）
- ❌ 不做"朋友圈"（避免社交表演）

## 技术假设（供讨论）

- 云端消息存储（换设备不丢记录）
- 多端同步协议（类似Telegram，非微信）
- 开源客户端，可选自托管服务器

## 参考竞品

| 产品 | 优点 | 缺陷 |
|------|------|------|
| Telegram | 多端同步、频道模式 | 国内访问困难、隐私争议 |
| Signal | 隐私极致 | 过于小众、功能简陋 |
| Discord | 社群管理强 | 过于复杂、游戏导向 |
| Slack | 异步友好 | 企业导向、成本高 |
